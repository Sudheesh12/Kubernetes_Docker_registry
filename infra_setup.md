# Infrastructure and Harbor Setup Guide

This document covers two main sections:
1. **Incident Resolution:** An explanation of the recent changes made to fix the Harbor Ingress issue.
2. **Infrastructure Bring-up Checklist:** A step-by-step guide with commands to deploy this entire stack from scratch (from Terraform up to Harbor).

---

## Part 1: What We Changed and Why

### The Original Problem
Initially, the Harbor deployment was configured to use the **AWS Application Load Balancer (ALB) Controller** (`ingressClassName: alb`) while simultaneously using **cert-manager** to provision a Let's Encrypt SSL certificate as a Kubernetes Secret (`harbor-tls`).

**Why did it fail?**
AWS ALBs terminate TLS connections natively at the load balancer level and strictily require certificates to be provisioned in **AWS Certificate Manager (ACM)**. They cannot read TLS certificates stored as standard Kubernetes Secrets. Because `cert-manager` was generating a Kubernetes Secret instead of an ACM certificate, the ALB controller endlessly errored out with:
`no certificate found for host: harbor.sudheeshbalakrishna.in`

### The Solution: Switching to NGINX Ingress Controller
To fix this without incurring additional AWS ACM complexities, we switched to the **NGINX Ingress Controller**. NGINX natively runs inside the cluster and is fully capable of reading standard Kubernetes Secrets (generated by `cert-manager`) to terminate TLS traffic.

**Changes Made:**
1. **Cleaned up ALB to avoid costs:** Deleted the broken `harbor-ingress` ALB resource to ensure AWS didn't charge for unused load balancers.
2. **Installed NGINX Ingress:** Deployed the `ingress-nginx` Helm chart. We explicitly configured its backing service to provision an **AWS Network Load Balancer (NLB)** in `internet-facing` mode. An NLB simply passes the TCP traffic (Ports 80/443) directly into the EKS cluster, allowing NGINX to handle the HTTP routing and TLS termination.
3. **Reconfigured Harbor:** Updated `harbor-values.yaml` to change `ingress.className` from `alb` to `nginx`. We also replaced the AWS-specific annotations with NGINX-specific ones (like proxy body size and ssl redirection).
4. **Upgraded Harbor:** Ran `helm upgrade` to apply the new NGINX ingress rules.
5. **DNS Update:** Updated the CNAME record for `harbor.sudheeshbalakrishna.in` to point to the new NGINX NLB DNS name instead of the old ALB.

## Part 2: Infrastructure Bring-Up Checklist

If you ever need to tear down and bring up the infrastructure from scratch, execute the following commands in order.

### 1. Provision Infrastructure with Terraform
Navigate to your Terraform directory to provision the VPC, IAM Roles, EKS Cluster, and Storage classes.
```bash
cd terraform/
terraform init
terraform plan
terraform apply -auto-approve
```

### 2. Configure Local `kubectl` Access
Once the cluster is up, configure your local environment to communicate with it.
*(Replace `<cluster_name>` with your actual EKS cluster name defined in Terraform variables)*
```bash
aws eks update-kubeconfig --region us-east-1 --name <cluster_name>
```

### 3. Deploy cert-manager
Cert-manager handles automatic SSL certificate provisioning via Let's Encrypt.
```bash
helm repo add jetstack https://charts.jetstack.io
helm repo update
helm install cert-manager jetstack/cert-manager \
  --namespace cert-manager \
  --create-namespace \
  --set installCRDs=true

# Wait a minute for pods to be ready, then apply your cluster issuer:
kubectl apply -f manifest/clusterissuer.yaml
```

### 4. Deploy NGINX Ingress Controller
This provisions the AWS Network Load Balancer (NLB) and handles all incoming HTTP/HTTPS traffic routing.
```bash
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo update
helm upgrade --install ingress-nginx ingress-nginx/ingress-nginx \
  --namespace ingress-nginx \
  --create-namespace \
  --set controller.service.annotations."service\.beta\.kubernetes\.io/aws-load-balancer-type"=nlb \
  --set controller.service.annotations."service\.beta\.kubernetes\.io/aws-load-balancer-scheme"=internet-facing
```

### 5. Deploy Harbor
Deploy the Harbor registry using your provided values file.
```bash
# Create the namespace if it doesn't exist
kubectl create namespace registryproject

helm repo add harbor https://helm.goharbor.io
helm repo update

helm install harbor harbor/harbor \
  --namespace registryproject \
  --version 1.14.0 \
  --timeout 10m \
  -f manifest/harbor-values.yaml
```

### 6. Update DNS Records
Once Harbor is deployed, you must map the NGINX Load Balancer's DNS name to your custom domain.
1. Get the NLB address:
   ```bash
   kubectl get svc ingress-nginx-controller -n ingress-nginx -o jsonpath='{.status.loadBalancer.ingress[0].hostname}'
   ```
2. Go to your DNS provider (e.g., Route 53 or GoDaddy).
3. Create a **CNAME** or **ALIAS** record for `harbor.sudheeshbalakrishna.in` pointing to the NLB hostname retrieved in the previous step.

Wait roughly 5-10 minutes for DNS propagation and for `cert-manager` to issue the certificate. You can then access Harbor at `https://harbor.sudheeshbalakrishna.in`.

## Part 3: Infrastructure Teardown Checklist

If you need to tear down the environment to stop incurring costs, run the following commands in order. **Do not just run `terraform destroy` immediately, as Kubernetes PVCs and Load Balancers created by Helm charts can prevent Terraform from cleanly destroying the VPC and subnets.**

### 1. Uninstall Harbor and Delete PVCs
```bash
helm uninstall harbor -n registryproject
# CRITICAL: Delete the Persistent Volume Claims so the underlying AWS EBS volumes are destroyed. 
# If you skip this, Terraform will hang trying to delete the VPC.
kubectl delete pvc --all -n registryproject
```

### 2. Uninstall NGINX Ingress Controller
```bash
helm uninstall ingress-nginx -n ingress-nginx
# This ensures the AWS Network Load Balancer (NLB) is deleted
kubectl delete namespace ingress-nginx
```

### 3. Uninstall Cert-Manager
```bash
helm uninstall cert-manager -n cert-manager
kubectl delete namespace cert-manager
```

### 4. Delete Harbor Namespace
```bash
kubectl delete namespace registryproject
```

### 5. Destroy Terraform Infrastructure
Now that the cluster-managed resources (NLBs, EBS volumes) are cleaned up, it is safe to destroy the AWS infrastructure.
```bash
cd terraform/
terraform destroy -auto-approve
```
